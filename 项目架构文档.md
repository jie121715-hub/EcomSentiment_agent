## 🎤 面试话术（2分钟讲清执行流程）

> 我做的是一个**电商智能客服 Agent 系统**，核心解决两个问题：**延迟高**和**安全弱**。
>
> 用户发来一条消息，先经过 **PerceptionAgent** — 用本地 BERT 做情感分析（7类）和意图识别（4类），同时 LLM 提取订单号、商品名等实体。
>
> 然后 **RouterAgent** 做三维决策：置信度不够就反问澄清；用户愤怒+投诉直接转人工；正常消息按 4 意图 → 4 Agent 一一对应分发。
>
> 80% 流量走 **KnowledgeQAAgent**，这里我做了一个**三层检索漏斗**来降延迟：第一层 Redis 精确缓存 `<5ms`，第二层 BM25 + MySQL FAQ 关键词匹配 `~15ms`，只有前两层都没命中才走第三层完整 RAG（父子块检索 + BGE-Reranker + LLM 生成）。加权平均从原来 3 秒降到 **400 毫秒**，提升 7 倍。
>
> 为防 LLM 瞎编政策，做了**高危门控**：检测到退款/法律类问题且知识库无精确文档 → 强制转人工。
>
> 另一个重点是**商户上传安全**：支持 PDF/DOCX 上传，先做 API Key 鉴权（Key 简化为 admin / merchant），再做内容安全扫描（18 种恶意模式），严重违规直接拒绝，可疑内容标记待审核。上传和 RAG 检索走同一个 ParentChildChunker，写入和读取格式一致。
>
> 🆕 Router 加了**商品咨询智能识别**：用户说"查下苹果18参数"，BERT 可能误判为业务处理（含"查"字），实际是商品咨询。新增 `_correct_product_inquiry_intent` 做二次修正：有商品词（手机/产品/参数…）+ 无业务词（物流/订单/退款…）→ 强制路由到 KnowledgeQA。FAQ 层也有 `_is_product_inquiry` 门控，商品咨询直接跳过 FAQ 走 LLM 通识兜底。
>
> 技术栈：FastAPI + LangGraph 编排 + BGE-M3 + Milvus + MySQL + Redis + DeepSeek API + PyMuPDF/python-docx。

## 一、Agent 体系

### 🎯 PerceptionAgent — 感知层
- **情感**: BERT 本地 2 分类(正/负, F1=89.5%) → LLM 7 种细粒度标签
- **意图**: BERT 10 分类 → LLM 置信度修正
- **实体**: LLM 提取 order_id / SKU / product_name / tracking_no
- **降级**: BERT 失败 → LLM 零样本兜底

### 🧠 RoutingAgent — 三维决策
| 维度 | 规则 |
|------|------|
| ① 置信度门控 | confidence < 50% → 反问澄清（订单号/快递单号/确认回复场景自动跳过）|
| ② 情绪紧急度 | ANGRY × (投诉\|退款) → CRITICAL → 转人工 + 自动建工单 |
| ③ 意图分发 | 4 意图 → 4 路 Agent（1:1 映射，见下表）|

**路由表**（4意图 → 4Agent，1:1基础映射 + 内容微调）:

| 意图 | 目标 Agent | 微调规则 |
|------|-----------|---------|
| knowledge_qa | **KnowledgeQA** | 三层检索（Redis→BM25→RAG） |
| business | **Business** | 含"政策/规则/流程"关键词 → 改写为 KnowledgeQA |
| knowledge_mgmt | **KnowledgeMgmt** | 商户指令，最高优先级 |
| escalate | **Escalate** | 含订单号 → 改写为 Business 优先处理 |

**🆕 商品咨询智能修正**（`_correct_product_inquiry_intent`）：
- 含商品词（手机/产品/参数/推荐…）+ 无业务词（物流/订单/退款…）+ 无ID格式 → 强制 routing 到 KnowledgeQA
- 典型场景："查下苹果18参数" — BERT 可能因"查"字误判 BUSINESS → 二次修正为 KNOWLEDGE_QA
- 与 FAQ 层门控（`_is_product_inquiry`）双重保障：Router 修正意图 + KnowledgeQA 跳过 FAQ

### 📚 KnowledgeQAAgent — 三层检索（核心）

```
问题 → ⚡Redis(<5ms) → 🔎BM25+FAQ(~15ms) → 🧬RAG(~2s) → 回复
```

| 层 | 技术 | 延迟 | 命中率 |
|----|------|------|--------|
| 第1层 | Redis 精确缓存 (answer:{query}) | <5ms | ~30% |
| 第2层 | BM25 + MySQL FAQ (jieba + BM25Okapi, raw_best/avg≥2.0+raw≥1.8) | ~15ms | ~50% |
| 🆕 | 商品咨询FAQ门控: _is_product_inquiry() 检测商品词→跳过FAQ→直走RAG | — | — |
| 第3层 | RAG: Query改写 → 父子块检索 → Reranker → LLM生成 | ~2s | ~20% |

**第3层 RAG 管线**（仅前两层未命中时触发）:
```
Query改写(4策略,可配置关闭) → Milvus子块检索 → parent_id去重
→ 父块1000字上下文 → BGE-Reranker精排 → LLM生成 → 异步写回Redis
```

**闲聊快速通道**: "你好"/"谢谢"→ 模板毫秒级 · 其他闲聊 → LLM 轻量生成

### 📦 BusinessAgent — 业务操作
- **读**: 🆕 查物流（订单号→DB订单状态；快递单号→DB关联订单→提示快递鸟即将上线）· 查订单 · 查库存
- **写**: 取消订单 · 退款 · 改地址（二次确认 + 上下文感知）

### 📝 KnowledgeMgmtAgent — 知识管理 + 文件上传
- **文本录入**: 商户对话式添加（触发词匹配）→ MySQL + Milvus 双写
- **🆕 文件上传**: `POST /admin/knowledge/upload` — PDF/DOCX → 文本提取 → 安全扫描 → ParentChildChunker 切分 → MySQL + Milvus 父子块双写

### 🚨 Escalate — 转人工
- 两种触发: ①情绪紧急(ANGRY+敏感意图) ②高危门控(政策法律无知识库验证)
- 自动创建工单 (TK-YYYYMMDD-XXXX) → 写入 `support_tickets`

## 七、面试高频问答

**Q: 三层检索怎么把延迟从3秒降到400ms的？**
> 客服场景80%是高频重复问题。旧版每条都走完整RAG管线（Query改写4次LLM+混合检索+Reranker+LLM生成），平均3秒。我加了两层快速通道：Redis精确缓存命中30%请求(<5ms)，BM25+MySQL FAQ关键词匹配命中50%请求(~15ms)。只有20%复杂问题才走完整RAG。加权平均从3秒降到408ms，提升7.4倍。BM25用jieba分词+raw_best/avg≥2.0+raw_best≥1.8双条件判断，23条FAQ已能正确区分命中/未命中。

**Q: 为什么用多Agent而不是一个LLM？**
> 单一LLM无法精确执行数据库读写、调用快递鸟API、做安全门控。我拆成5个Agent：Perception管感知、Router做三维决策、KnowledgeQA管语义理解（三层检索）、Business管精确事务（MySQL读写+快递鸟API）、KnowledgeMgmt管知识录入。解耦后可独立升级、独立测试。

**Q: Router三维决策怎么设计的？**
> 维度1置信度门控（<50%反问澄清；但订单号/快递单号/确认回复场景自动跳过）；维度2情绪紧急度（愤怒+敏感意图→CRITICAL转人工）；维度3意图分发（4意图→4路Agent，1:1映射）。优先级：紧急接管 > 反问 > 正常分发。

**Q: 父子块检索是怎么回事？**
> 传统等分切块会导致上下文碎片化。我的方案：ParentChildChunker把每条知识切成父块1000字（给LLM看的完整上下文）和子块300字（用于向量检索）。搜索时只匹配子块（精准），通过parent_id去重后取父块（完整），LLM拿到的是1000字连贯上下文而非碎片。

**Q: 高危门控是做什么的？**
> LLM会瞎编数字承诺——比如你问"假一赔几"，它可能编造"假一赔十"。高危门控检测到退款/法律类关键词时，先在知识库搜精确政策文档：有→基于文档回答，无→强制转人工，绝不用LLM通识编造。

**Q: 怎么防止用户上传恶意政策（比如假一赔十）？**
> 三道防线：① API Key鉴权——只有merchant/admin能上传，普通用户403拒绝；② 内容安全扫描——18种恶意模式匹配，严重违规（假一赔十/百倍赔偿）直接422拒绝，可疑内容（官方授权/品牌直营）标记pending_review；③ 高危门控——即使入库了，RAG生成侧仍会在回答前拦截。

**Q: 上传的PDF/DOCX怎么保证和RAG检索一致？**
> DocLoader提取文本后，和RAG管线用同一个ParentChildChunker切分（父块1000字+子块300字），子块带chunk_type="child"+parent_id写入Milvus。写入和读取走同一套数据结构，保证格式一致。

**Q: 多轮对话中的确认操作怎么处理？**
> 上下文感知确认：上轮发了"确定要取消吗？请回复确认"→本轮用户说"取消"→Router检测到确认场景跳过低置信度反问→BusinessAgent从历史提取order_id→直接执行。不需要用户重复提供订单号。
