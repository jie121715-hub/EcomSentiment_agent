🛡️ 4.5. 安全边界 — 高危门控 + LLM兜底策略

🚨 高危门控（政策/法律/赔付）
30+ 高危关键词: 退款 退货 换货 赔付 赔偿 假一赔 赔几 法律 条款 保修 质保 三包 假货 真伪 维权 12315 发票 消费者权益

门控逻辑:
① 意图检测(after_sales/complaint) + 关键词匹配 → 高危判定
② 检查RAG检索的文档是否真的包含相关政策内容
③ 无精确匹配 → 🚨 强制转人工，绝不用LLM通识编造

案例: "假一赔几?" → 知识库无此政策 → 转人工，不瞎编"假一赔十"
🤖 LLM通识兜底（非高危场景）
触发条件: 商品参数/通识问题 + RAG搜不到
Few-shot 自然免责:
❌ 旧: "💡以上信息基于通用知识，如需了解本店具体商品详情，建议查看商品页面或联系客服400-618-8888"
✅ 新: "这款手机一般配6000mAh电池，正常用一天半没问题～不过具体续航还是以咱们商品页为准哦！"

Prompt 策略: 用正反示例教会LLM把免责自然融进句子里，读起来像真人客服
📊 安全边界矩阵
场景知识库有知识库无
商品参数/通识✅ 基于文档✅ LLM兜底+自然免责
退款政策/法律(高危)✅ 基于文档🚨 强制转人工
假货赔付/保修(高危)✅ 基于文档🚨 强制转人工
业务操作(取消/退款)—✅ BusinessAgent+确认
💬 上下文感知确认 (v3.2)
问题: "取消订单JDxxx"→确认提示→用户说"取消"→系统反问"您想做什么?"
修复:
① Router 检测上轮是确认提示 → 跳过置信度反问
② BusinessAgent 从历史提取 order_id → 无需重复提供
③ 简短回复("取消"/"是的"/"行")→识别为确认→执行

状态检测: 上轮含"确定要…请回复确认" → 本轮自动走确认通道
